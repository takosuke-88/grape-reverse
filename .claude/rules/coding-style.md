# Coding Style & UI/UX Rules

## 1. 言語・スタイリング基準
- **言語**: TypeScript (型安全性を担保、型チェック `npx tsc` が通るコード)。
- **スタイリング**: Tailwind CSS を全面的に使用。インラインスタイルは原則禁止。

## 2. 現場至上主義のUI/UX設計
- **巨大なタップ領域**: ホールで台を打ちながら片手でブラインドタッチができるよう、ボタンやカウンターには巨大なタップ領域を確保する。視認性よりも「手の感覚」での操作性を優先する。
- **画面分割（スプリットビュー）適応**: スマホの上半分で動画視聴、下半分でツールを操作する状況を想定し、縦幅が狭くなっても崩れない・操作性を損なわないレスポンシブ設計を維持する。
- **破壊的操作の防御**: 全体リセットなどの破壊的アクションボタンは、通常の入力導線から完全に隔離した位置に配置し、鮮やかな強い赤（`bg-red-600 text-white font-semibold`）で視認性を高め、必ず `window.confirm` による確認ダイアログを挟む。

## 3. カウンターコンポーネント設計 (30:70スプリット)
- **構造**: `DynamicInput.tsx` 等の共通カウンターUIは、左側30%を数値・操作領域、右側70%をプラス（＋1）エリアとする。
- **操作仕様**:
  - **数字部分（左側30%領域）**: 数字部分を**ワンタップ**した瞬間に、直接入力モード（テンキー・キーパッド）が一発で即座に起動する。
  - **プラス領域（右70%領域）**: タップで単一加算（+1）。バー内部には総ゲーム数から算出した確率（例: `1/128.5`）を**白の斜体（italic）**で表示する。ゲーム数が0の時は `-` を表示する。不要な小さな補助数字は排除すること。
  - **マイナス値の防止（ゼロガード）**: 全てのカウンターにおいて、数値が `0` の時に「−」ボタンを押してもマイナスにならないよう制御する。

## 4. ターゲット端末 & レスポンシブ挙動
- **ターゲット解像度**:
  - iPhone 14系（390×844px）/ iPhone XR・11（414×896px）
  - Android（412×915px）
  - 最小幅: 375px（iPhone SE）※画面が崩れないこと
- **ヘッダーレスポンシブ**:
  - 左側のブランド名「Grape Reverse」は、画面幅が狭くなって右側のメニュー項目（ジャグラー、ハナハナ、攻略コラム）と衝突しそうになった場合、メニューを下に回さず、ロゴ自体を縦2行（`Grape` と `Reverse`）に自動改行させる。
  - 右側メニューは `white-space: nowrap` で一とかたまりを維持する。
  - 2行に改行した際にすぐ下の要素と被らないよう、レイアウト構造で下へ押し出す（`gap-y-2`等）。
- **Stickyヘッダー仕様**:
  - 画面を下にスクロールした際、最上部のロゴ・メニューバーは画面外へスクロールアウトさせる。
  - その下の「機種名バー（マゼンタ色）」と「ナビゲーションバー（グレー）」の2行セットだけを、画面最上部に常に追従（`sticky top-0 z-50`）させる。

## 5. バイブレーション実装
- iOS Safariは `navigator.vibrate()` 未対応のため、必ず `const supportsVibration = 'vibrate' in navigator;` で環境分岐を行う。
- バイブのオン/オフトグルUIは、未対応端末で丸ごと非表示にすると機能の存在が認知できないため、UIは表示しつつ、タップ時に未対応である旨を通知するか、適切なフォールバック（Web Audio等）を検討する（非対応だからといってヘッダーから消去しない）。デフォルトはオフ。
- バイブONの時、トグルがONになった瞬間は短く鋭いバイブを1回鳴らす。
